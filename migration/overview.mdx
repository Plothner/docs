Quanterios automates cryptographic migration from planning through deployment. Select from 29 supported migration paths, generate runnable infrastructure code, predict breakage probability before deploying, and deliver the migration as a pull request to your repository. These capabilities are powered by Quanterios Migration Intelligence.

The migration workflow connects directly to the risk assessment stage: every weak algorithm, expiring certificate, and compliance gap surfaced during discovery can be converted into a runnable fix. You can also specify any source and target algorithm manually, without requiring a prior discovery scan.

## 29 Migration Paths

Quanterios ships with 29 pre-built migration paths across three categories:

| Category | Count | Purpose |
|---|---|---|
| Classical | 13 | Fix today's compliance and hygiene gaps |
| Quantum | 10 | Prepare for post-quantum cryptography by 2030 |
| Hybrid | 6 | Deploy classical and PQC simultaneously for zero-downtime transition |

See [Migration Paths](/migration/paths) for the full table.

## End-to-End Workflow

<Steps>
  <Step title="Select a migration path">
    Choose from 29 available templates. Each template encodes the source algorithm, target algorithm, platform, and IaC type (Terraform, Ansible, or Helm).
  </Step>
  <Step title="Predict breakage before deploying">
    Before any plan is created, an XGBoost model scores your specific migration on a 0 to 100% breakage probability scale. Inputs include library version, dependency count, platform, and historical outcomes from 5,000+ real GitHub crypto PRs. AUC 0.94.
  </Step>
  <Step title="Generate the playbook">
    The engine renders Jinja2 templates into fully parameterized HCL, YAML playbooks, or Helm values files. The rendered output is stored in the migration plan and version-controlled.
  </Step>
  <Step title="Patch application code">
    A Python code patcher applies hybrid shims to your codebase, letting old and new algorithms run side by side during the transition period. No manual editing required.
  </Step>
  <Step title="Deploy with canary rollout">
    Deployment is staged. The canary phase routes a configurable percentage of traffic to the new configuration. Metrics are collected for the full monitoring window before full rollout proceeds.
  </Step>
  <Step title="Monitor and confirm">
    Log-based monitoring tracks handshake ratios, cipher suite negotiations, and error rates throughout the monitoring window. When rolloff criteria are met, Quanterios opens a cleanup PR to remove the old configuration.
  </Step>
</Steps>
