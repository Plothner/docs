## The workflow in context

Risk Intelligence surfaces every weak algorithm, expiring certificate, and compliance gap in your environment. Migration Intelligence picks up from there: it converts each finding into a runnable fix with a predicted breakage probability before a single change reaches production.

The two products are designed to be used together, but Migration Intelligence accepts manual input as well. You can specify any source and target algorithm directly.

## 29 migration paths

Quanterios ships with 29 pre-built migration paths across three categories:

## End-to-end workflow

<Steps>
  <Step title="Select a template">
    Choose a migration path from the 29 available templates. Each template encodes the source algorithm, target algorithm, platform, and IaC type (Terraform, Ansible, or Helm).
  </Step>
  <Step title="Preview with breakage prediction">
    Before any plan is created, the XGBoost model scores your specific migration on a 0 to 100% breakage probability scale. Inputs include library version, dependency count, platform, and historical outcomes from 5,000+ real GitHub crypto PRs.
  </Step>
  <Step title="Create the plan">
    The engine renders Jinja2 templates into fully-parameterized HCL, YAML playbooks, or Helm values files. The rendered output is stored in the migration plan and version-controlled.
  </Step>
  <Step title="Deploy: canary then rollout">
    Deployment is staged. The canary phase routes a configurable percentage of traffic to the new configuration. Metrics are collected for the full monitoring window before full rollout proceeds.
  </Step>
  <Step title="Monitor">
    Log-based monitoring tracks handshake ratios, cipher suite negotiations, and error rates throughout the monitoring window. No agent required.
  </Step>
  <Step title="Confirm and close">
    When rolloff criteria are met, Quanterios opens a cleanup PR to remove the old configuration. The migration moves to a confirmed state in the portfolio.
  </Step>
</Steps>
