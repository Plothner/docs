The fastest way to start with Quanterios is the Crypto Debt Firewall. Add a single GitHub Action to your repository and every pull request will be scanned for cryptographic anti-patterns before it merges. No infrastructure to provision, no agent to install.

<Info>
  This guide covers GitHub. If you use GitLab or Bitbucket, the setup is equivalent but uses platform-specific CI syntax. See the [GitLab integration guide](/customer/integrations/gitlab) and the [Bitbucket integration guide](/customer/integrations/bitbucket) for step-by-step instructions.
</Info>

## Prerequisites

Before you begin, confirm you have the following:

- A GitHub repository where you want to enable scanning
- Admin access to the repository (required to add Actions and configure secrets)

## Install the Firewall

<Steps>
  <Step title="Add the GitHub Action">
    Create the file `.github/workflows/quanterios-firewall.yml` in your repository with the following content. Commit it directly to the default branch so it takes effect immediately for all subsequent pull requests.

    ```yaml .github/workflows/quanterios-firewall.yml
    name: Quanterios Crypto Debt Firewall

    on:
      pull_request:
        branches:
          - main
          - master

    permissions:
      contents: read
      pull-requests: write

    jobs:
      firewall:
        name: Scan for cryptographic anti-patterns
        runs-on: ubuntu-latest

        steps:
          - name: Checkout repository
            uses: actions/checkout@v4
            with:
              fetch-depth: 0

          - name: Run Quanterios Firewall
            uses: Plothner/firewall-action@v1
            with:
              backend_url: https://api.quanterios.com
              github_token: ${{ secrets.GITHUB_TOKEN }}
    ```

    The `GITHUB_TOKEN` is provided automatically by GitHub Actions. No additional secrets are required to get started.
  </Step>

  <Step title="Open a Pull Request">
    With the workflow committed, open any pull request against your default branch. The Quanterios Firewall job will run automatically as part of the pull request checks.

    To see a finding immediately, open a pull request that includes code like the following. The scanner will flag the RSA key size and post a finding.

    ```python
    from cryptography.hazmat.primitives.asymmetric import rsa

    # This will be flagged: RSA-1024 is below the minimum recommended key size
    private_key = rsa.generate_private_key(
        public_exponent=65537,
        key_size=1024,
    )
    ```

    The Action status will appear in the pull request checks panel alongside your existing CI jobs. A passing check means no findings above the confidence threshold were detected. A failing check means at least one finding was posted for review.
  </Step>

  <Step title="Review Findings">
    When the scanner detects a cryptographic anti-pattern, it posts a structured review comment directly on the pull request. The comment contains two sections.

    **Semgrep findings** list each matched pattern with the file path, line number, rule ID, and a description of the risk. These are deterministic matches based on the Quanterios rule set.

    **AI auditor analysis** interprets the findings in context. The LLM auditor reads the surrounding code, assesses whether the pattern is a genuine risk or a false positive, explains the HNDL relevance of each finding, and recommends the specific remediation step. Each finding includes a confidence score between 0.0 and 1.0.

    ```
    ## Quanterios Crypto Debt Firewall

    ### Semgrep Findings

    | Severity | File | Line | Rule | Description |
    |---|---|---|---|---|
    | HIGH | src/auth/keys.py | 12 | CDF-RSA-001 | RSA key size below 2048 bits |

    ### AI Auditor Analysis

    **Finding: RSA-1024 key generation (confidence: 0.91)**

    This key is used to sign authentication tokens issued to external clients.
    RSA-1024 has been considered broken since 2010 and is not resistant to
    classical attacks, let alone quantum attacks. Migrate to RSA-4096 or,
    preferably, to an ML-DSA-based signing scheme. A migration playbook for
    this pattern is available in Migration Intelligence.
    ```
  </Step>
</Steps>

## What Happens Next

The firewall uses a confidence threshold to decide how findings are surfaced.

| Confidence | Action |
|---|---|
| High (>= 0.6) | Finding is posted as a pull request review comment and the check fails |
| Low (< 0.6) | Finding is recorded as a training signal and the check passes silently |

Low-confidence findings are not discarded. They are used to improve the AI auditor model over time. As the model improves, previously ambiguous patterns may cross the threshold and become active findings.

<Tip>
  Developers can leave a thumbs up or thumbs down reaction on any firewall comment to provide feedback directly from GitHub. Thumbs up confirms the finding is valid. Thumbs down marks it as a false positive. Both signals are ingested by Quanterios and used to refine the confidence model for your codebase specifically.
</Tip>
