<Note>
  The PQC CA complements the existing migration playbooks. Today, quantum migration paths (`pqc-*` and `hyb-*` templates) source ML-KEM and ML-DSA certificates from external CAs. The PQC CA replaces that external dependency with a fully managed, Quanterios-operated issuer that integrates directly with the migration pipeline.
</Note>

## The problem it solves

External certificate authorities are not yet issuing ML-KEM and ML-DSA certificates at scale. Organizations running quantum or hybrid migrations today must either:

- Operate their own ACME-compatible CA with OQS support, which requires significant infrastructure and cryptographic expertise, or
- Use self-signed certificates in non-production environments and wait for commercial CA support.

The Quanterios PQC CA eliminates both workarounds. It issues FIPS 203 (ML-KEM) and FIPS 204 (ML-DSA) certificates as a managed service, with the same lifecycle automation Quanterios already provides for classical certificates.

## Lifecycle management

| Event | Automated action |
|---|---|
| Certificate issued | Recorded in CBOM, linked to assets discovered by Risk Intelligence |
| 30 days before expiry | Renewal initiated automatically if ACME configured |
| 14 days before expiry | Alert sent to tenant admin if renewal has not completed |
| 7 days before expiry | Escalation alert; manual renewal link provided |
| Certificate revoked | OCSP responder updated within 5 minutes; affected assets flagged in Risk Intelligence |
| Certificate rotated | Rotation playbook PR opened on configured platform |

## Integration with Migration Intelligence

When the PQC CA is available, quantum and hybrid migration templates will default to issuing certificates via the Quanterios PQC CA rather than an external issuer. No changes to the migration workflow are required. The template parameter `issuer` will accept `quanterios-pqc` as a value alongside the existing external CA options.
