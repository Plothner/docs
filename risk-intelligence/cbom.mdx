## What a CBOM is

A Cryptographic Bill of Materials is an exhaustive inventory of every cryptographic primitive, protocol, and asset your organization uses. Where a software SBOM lists libraries and dependencies, a CBOM lists certificates, keys, algorithms, and the systems that depend on them.

Quanterios builds and maintains your CBOM automatically from discovery scans. It is the single source of truth for all downstream capabilities: HNDL scoring, compliance gap analysis, migration planning, and board reports.

## Asset types

| Type | Description | Example sources |
|---|---|---|
| `tls_cert` | X.509 certificates used for TLS, including validity period, SANs, issuer chain, and key algorithm | ACM, Azure Key Vault, cert-manager, EJBCA |
| `cloud_key` | Encryption keys managed by cloud KMS services, including algorithm, key spec, and rotation schedule | AWS KMS, Azure Key Vault, Cloud KMS |
| `ssh_key` | SSH public keys, including algorithm, key size, authorized hosts, and last-used metadata | IAM access keys, LDAP, on-prem inventory |
| `code_crypto_call` | Cryptographic API calls detected in source code, including file, line, function, and algorithm | Semgrep code scanner |

## Asset metadata fields

Every CBOM asset carries a common set of metadata fields regardless of type:

| Field | Type | Description |
|---|---|---|
| `id` | UUID | Stable asset identifier |
| `asset_type` | enum | `tls_cert`, `cloud_key`, `ssh_key`, `code_crypto_call` |
| `algorithm` | string | Cryptographic algorithm (e.g. `RSA-2048`, `ECDSA-P256`, `AES-256`) |
| `source_connector` | string | The connector that discovered this asset |
| `source_id` | string | The platform-native identifier (ARN, Key Vault URI, etc.) |
| `discovered_at` | ISO 8601 | When the asset was first discovered |
| `last_seen_at` | ISO 8601 | Most recent confirmation the asset still exists |
| `hndl_score` | integer (0-100) | Current HNDL risk score |
| `compliance_gaps` | array | List of framework IDs where this asset is non-compliant |
| `tags` | object | Key-value pairs from the source platform |

## Asset deduplication

The same physical asset can appear across multiple connectors. For example, an ACM certificate may also surface in CloudFront and ELB connectors. Quanterios deduplicates by **fingerprint**: SHA-256 of the certificate DER encoding for `tls_cert`, platform-native key ID for `cloud_key`, and SHA-256 of the public key for `ssh_key`.

Duplicates are merged into a single CBOM record with all source references listed under `source_refs`.

## Bulk import via API

If you have an existing asset inventory (from a previous tool or a manual audit), you can bulk-import assets via the CBOM API. Imported assets are merged with discovered assets using the same deduplication logic.

```bash
curl -X POST https://api.quanterios.com/v1/cbom/bulk \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"assets": [...]}'
```

See the [CBOM API Reference](/api-reference/cbom) for the full schema and bulk import limits.
