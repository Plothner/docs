    ## Ingest Semgrep findings

    `POST /api/v1/firewall/scan`

    Ingest an array of Semgrep findings for a pull request. Quanterios creates or updates cryptographic assets from each finding and returns counts for observability.

    ### Request body

    <ParamField body="findings" type="array" required>
      Array of Semgrep finding objects. Each element must include `rule_id`, `path`, `start.line`, and `extra.message`.
    </ParamField>

    <ParamField body="repo_owner" type="string" required>
      Owner or organization name of the repository (for example, `acme-corp`).
    </ParamField>

    <ParamField body="repo_name" type="string" required>
      Repository name without the owner prefix (for example, `payments-service`).
    </ParamField>

    <ParamField body="pr_number" type="integer" required>
      Pull-request number the findings belong to.
    </ParamField>

    <ParamField body="commit_sha" type="string" required>
      Full 40-character commit SHA that was scanned.
    </ParamField>

    ### Response

    <ResponseField name="ingested" type="integer">
      Total number of findings ingested from the request body.
    </ResponseField>

    <ResponseField name="assets_created" type="integer">
      Number of net-new cryptographic assets created from the ingested findings.
    </ResponseField>

    <CodeGroup>
    ```bash curl
    curl --request POST \
      --url https://api.quanterios.com/api/v1/firewall/scan \
      --header "X-Tenant-Id: ten_abc123" \
      --header "X-API-Key: qk_live_xxxxxxxxxxxx" \
      --header "Content-Type: application/json" \
      --data '{
        "findings": [
          {
            "rule_id": "crypto.rsa-key-too-short",
            "path": "src/auth/jwt.py",
            "start": { "line": 42 },
            "extra": { "message": "RSA key size 1024 is below the recommended 2048 bits." }
          }
        ],
        "repo_owner": "acme-corp",
        "repo_name": "payments-service",
        "pr_number": 317,
        "commit_sha": "a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4e5f6a1b2"
      }'
    ```
    ```json Response
    {
      "ingested": 1,
      "assets_created": 1
    }
    ```
    </CodeGroup>

    ## Run LLM audit

    `POST /api/v1/firewall/audit`

    Submit raw diff hunks for an LLM-powered cryptographic audit. The model inspects each hunk for weak algorithms, misconfigurations, and harvest-now-decrypt-later exposure, returning structured findings at two severity tiers.

    ### Request body

    <ParamField body="hunks" type="array" required>
      Array of diff hunk objects. Each object must contain `file` (string, relative path) and `diff` (string, unified-diff text).
    </ParamField>

    <ParamField body="repo_owner" type="string" required>
      Owner or organization of the repository.
    </ParamField>

    <ParamField body="repo_name" type="string" required>
      Repository name.
    </ParamField>

    <ParamField body="pr_number" type="integer" required>
      Pull-request number the diff belongs to.
    </ParamField>

    ### Response

    <ResponseField name="high_findings" type="array">
      Critical or high-severity findings. Each element includes `id`, `category`, `confidence`, `text`, `file`, and `line_hint`.
    </ResponseField>

    <ResponseField name="low_findings" type="array">
      Informational or low-severity findings with the same schema as `high_findings`.
    </ResponseField>

    <CodeGroup>
    ```bash curl
    curl --request POST \
      --url https://api.quanterios.com/api/v1/firewall/audit \
      --header "X-Tenant-Id: ten_abc123" \
      --header "X-API-Key: qk_live_xxxxxxxxxxxx" \
      --header "Content-Type: application/json" \
      --data '{
        "hunks": [
          {
            "file": "src/tls/config.py",
            "diff": "@@ -10,6 +10,6 @@\n-cipher = \"DES-CBC\"\n+cipher = \"AES-256-GCM\""
          }
        ],
        "repo_owner": "acme-corp",
        "repo_name": "payments-service",
        "pr_number": 317
      }'
    ```
    ```json Response
    {
      "high_findings": [
        {
          "id": "fnd_h_001",
          "category": "weak_cipher",
          "confidence": 0.97,
          "text": "DES-CBC is broken. Migrate to AES-256-GCM or ChaCha20-Poly1305.",
          "file": "src/tls/config.py",
          "line_hint": 10
        }
      ],
      "low_findings": []
    }
    ```
    </CodeGroup>

    ## Claim anonymous tenant

    `POST /api/v1/firewall/claim/{fingerprint}`

    Transfer all assets created under an anonymous session fingerprint to an authenticated tenant. Use this after a user signs up following a frictionless first scan.

    ### Path parameters

    <ParamField path="fingerprint" type="string" required>
      Anonymous session fingerprint returned during the unauthenticated scan.
    </ParamField>

    ### Headers

    | Header | Required | Description |
    |--------|----------|-------------|
    | `Authorization` | Yes | `Bearer <access_token>` |

    ### Response

    <ResponseField name="tenant_id" type="string">
      The authenticated tenant that now owns the transferred assets.
    </ResponseField>

    <ResponseField name="assets_transferred" type="integer">
      Number of assets moved from the anonymous session to the tenant.
    </ResponseField>

    <CodeGroup>
    ```bash curl
    curl --request POST \
      --url https://api.quanterios.com/api/v1/firewall/claim/fp_anon_7f3a9c \
      --header "Authorization: Bearer eyJhbGciOiJSUzI1NiJ9..."
    ```
    ```json Response
    {
      "tenant_id": "ten_abc123",
      "assets_transferred": 4
    }
    ```
    </CodeGroup>

    ## GitHub App webhook

    `POST /api/v1/firewall/webhook/github`

    Receive GitHub App events. Quanterios validates the signature, then triggers scans or posts status checks based on the event type.

    ### Headers

    | Header | Required | Description |
    |--------|----------|-------------|
    | `X-Hub-Signature-256` | Yes | HMAC-SHA256 signature of the raw request body, prefixed with `sha256=`. |
    | `X-GitHub-Event` | Yes | GitHub event name, for example `pull_request` or `issue_comment`. |

    ### Request body

    The raw GitHub event payload as delivered by GitHub. Quanterios does not transform the payload before ingestion.

    <CodeGroup>
    ```bash curl
    curl --request POST \
      --url https://api.quanterios.com/api/v1/firewall/webhook/github \
      --header "X-Hub-Signature-256: sha256=abc123..." \
      --header "X-GitHub-Event: pull_request" \
      --header "Content-Type: application/json" \
      --data '{ "action": "opened", "number": 317, "repository": { "full_name": "acme-corp/payments-service" } }'
    ```
    ```json Response
    {
      "received": true
    }
    ```
    </CodeGroup>

    ## GitLab webhook

    `POST /api/v1/firewall/webhook/gitlab`

    Receive GitLab merge-request and pipeline events. Quanterios validates the secret token before processing.

    ### Headers

    | Header | Required | Description |
    |--------|----------|-------------|
    | `X-Gitlab-Token` | Yes | The secret token configured in your GitLab webhook settings. |

    ### Request body

    The raw GitLab event payload as delivered by GitLab.

    <CodeGroup>
    ```bash curl
    curl --request POST \
      --url https://api.quanterios.com/api/v1/firewall/webhook/gitlab \
      --header "X-Gitlab-Token: glwh_secret_token" \
      --header "Content-Type: application/json" \
      --data '{ "object_kind": "merge_request", "project": { "id": 99 }, "object_attributes": { "iid": 42, "action": "open" } }'
    ```
    ```json Response
    {
      "received": true
    }
    ```
    </CodeGroup>

    ## Start GitLab OAuth

    `GET /api/v1/firewall/gitlab/oauth/start`

    Begin the GitLab OAuth 2.0 authorization flow. The response is a redirect to `gitlab.com`; direct the user's browser to this endpoint rather than calling it from a server.

    ### Query parameters

    <ParamField query="project_id" type="string" required>
      GitLab numeric project ID that the OAuth grant will cover.
    </ParamField>

    ### Response

    `302 Found` redirect to the GitLab authorization page. After the user grants access, GitLab redirects back to the Quanterios callback URL with the authorization code.

    <CodeGroup>
    ```bash curl
    curl --request GET \
      --url "https://api.quanterios.com/api/v1/firewall/gitlab/oauth/start?project_id=99" \
      --header "X-Tenant-Id: ten_abc123" \
      --header "X-API-Key: qk_live_xxxxxxxxxxxx" \
      --verbose
    ```
    ```text Response
    HTTP/2 302
    location: https://gitlab.com/oauth/authorize?client_id=...&redirect_uri=...&scope=api
    ```
    </CodeGroup>

    ## Start Bitbucket OAuth

    `GET /api/v1/firewall/bitbucket/oauth/start`

    Begin the Bitbucket OAuth 2.0 authorization flow. Redirect the user's browser to this endpoint; do not call it server-side.

    ### Query parameters

    <ParamField query="workspace" type="string" required>
      Bitbucket workspace slug that the OAuth grant will cover (for example, `acme-corp`).
    </ParamField>

    ### Response

    `302 Found` redirect to the Bitbucket authorization page.

    <CodeGroup>
    ```bash curl
    curl --request GET \
      --url "https://api.quanterios.com/api/v1/firewall/bitbucket/oauth/start?workspace=acme-corp" \
      --header "X-Tenant-Id: ten_abc123" \
      --header "X-API-Key: qk_live_xxxxxxxxxxxx" \
      --verbose
    ```
    ```text Response
    HTTP/2 302
    location: https://bitbucket.org/site/oauth2/authorize?client_id=...&redirect_uri=...
    ```
    </CodeGroup>
